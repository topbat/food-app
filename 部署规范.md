# 分支与部署规范

## 一、分支模型（共三个分支）

| 分支 | 角色 | 对应环境 | 说明 |
|---|---|---|---|
| `dev` | 日常开发分支 | **dev** | 功能开发与联调都在此分支进行，推送即自动部署 dev 环境 |
| `master` | 预发布分支 | **uat** | 从 `dev` 合并而来，用于 UAT 验收测试 |
| `main` | 生产分支（默认分支） | **prod** | 仅从 `master` 合并，对应线上生产环境 |

**合并流向（单向，禁止跳级）**：

```
dev ──(功能完成，发起合并)──▶ master ──(UAT 验收通过)──▶ main
```

- 日常开发提交到 `dev`；禁止直接向 `master` / `main` 提交业务代码
- 每次合并/提交必须同步更新 [CHANGELOG.md](CHANGELOG.md)（见提交规范）
- 紧急修复（hotfix）：从 `main` 拉出修复后，需同时回合到 `master` 与 `dev`

## 二、部署流水线（.github/workflows/deploy.yml）

流水线包含 5 个任务：解析目标环境 → 并行【后端 Maven 打包 / 前端 npm 构建 / AI 服务自检】→ 部署。
构建产物（后端 jar、前端 dist）以 Artifact 形式保留 7 天，可在运行详情页下载。

> 当前「部署」步骤为模拟发布（echo 输出发布清单）。接入真实服务器时：
> 在仓库 **Settings → Secrets and variables → Actions** 配置服务器地址/SSH 密钥，
> 替换 deploy.yml 最后一步为 scp/ssh 或容器发布命令即可，分支→环境映射逻辑无需改动。

### 触发方式一：分支推送自动触发

```powershell
git push origin dev      # 自动触发 → 部署 dev 环境
git push origin master   # 自动触发 → 部署 uat 环境
git push origin main     # 自动触发 → 部署 prod 环境
```

### 触发方式二：脚本触发（gh CLI，前置 gh auth login）

仓库内提供封装脚本 [scripts/deploy.ps1](scripts/deploy.ps1)：

```powershell
cd d:\self\food-app
.\scripts\deploy.ps1                               # 默认：dev 分支 → dev 环境
.\scripts\deploy.ps1 -Branch master                # master → uat（自动映射）
.\scripts\deploy.ps1 -Branch main                  # main → prod（自动映射）
.\scripts\deploy.ps1 -Branch main -Environment uat # 显式覆盖环境映射
.\scripts\deploy.ps1 -Branch dev -Watch            # 触发后实时跟踪运行日志
```

也可直接使用原生 gh 命令：

```powershell
gh workflow run deploy.yml --repo topbat/food-app --ref dev  -f environment=auto   # dev → dev
gh workflow run deploy.yml --repo topbat/food-app --ref main -f environment=prod   # main → prod
gh run list  --repo topbat/food-app --workflow=deploy.yml --limit 5                # 查看运行记录
gh run watch --repo topbat/food-app                                                # 跟踪最近一次运行
```

### 触发方式三：GitHub 网页手动触发

1. 打开仓库页：<https://github.com/topbat/food-app>
2. 点顶部 **Actions** 标签页（路径：`https://github.com/topbat/food-app/actions`）
3. 左侧 Workflow 列表选择 **deploy**
4. 右侧点击 **Run workflow** 灰色下拉按钮
5. 在下拉面板中：
   - **Use workflow from** 选择分支（`dev` / `master` / `main`）
   - **目标环境** 下拉选择 `auto`（按分支自动映射）或显式指定 `dev` / `uat` / `prod`
6. 点击绿色 **Run workflow** 按钮，几秒后页面出现新的运行记录，点进去可看各任务实时日志与产物下载

## 三、环境对照表

| 环境 | 后端 profile | AI 服务 | 数据库 | 用途 |
|---|---|---|---|---|
| dev | `--spring.profiles.active=dev` | `APP_ENV=dev` | H2 内存库（免安装） | 开发联调 |
| uat | `--spring.profiles.active=uat` | `APP_ENV=uat` | MySQL `food_app` | 验收测试 |
| prod | `--spring.profiles.active=prod` | `APP_ENV=prod` | MySQL `food_app` | 生产 |
