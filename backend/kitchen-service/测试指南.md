# 厨房引擎服务（kitchen-service）测试指南

## 1. 模块简介

| 项目 | 说明 |
|---|---|
| 服务名 | kitchen-service（厨房引擎服务，高频服务） |
| 端口 | **8083** |
| 接口前缀 | `/api/kitchen`（**全部接口需鉴权**，无白名单） |
| 表前缀 | `kitchen_`（kitchen_session / kitchen_timer / kitchen_voice_log） |
| 核心职责 | 烹饪会话五步法状态机（PREPARE→WASH→CUT→COOK→PLATE）、多任务并行倒计时、语音指令解析 |
| 技术 | Spring Boot 3.3.5 + Java 21 + JPA，启动类开启 `@EnableScheduling`（计时器到点每5秒兜底扫描） |

**对其他服务的依赖（启动顺序）**：

1. **菜谱服务 :8082（强依赖，必须先启动）**——创建会话时调用 `GET /api/recipe/{id}`（取菜谱名 `data.info.title`）与 `GET /api/recipe/{id}/steps`（取步骤平铺列表）；步骤列表在本服务内存缓存 **10 分钟**（ConcurrentHashMap）。菜谱服务不可用时创建/查询会话返回 **50200**。
2. **用户服务 :8081（间接依赖）**——本服务不直接调用用户服务，但所有接口需要 JWT Token，需先到用户服务注册/登录获取 Token（dev 环境各服务共享同一默认 JWT 密钥，签发后可直接在本服务使用）。

> 推荐启动顺序：菜谱服务 8082 → 用户服务 8081 → 厨房服务 8083。

进度算法（与契约一致）：阶段权重 PREPARE=10 / WASH=5 / CUT=15 / COOK=60 / PLATE=10，`进度 = 已走完阶段权重之和 + 当前阶段权重 × (currentStepIndex-1) / 当前阶段步骤数`，已完成会话恒为 100。

## 2. 环境准备

### 2.1 dev（默认，免安装）

- H2 内存库（MySQL 兼容模式），`ddl-auto: update` 自动建表，**零外部依赖**。
- JDBC URL：`jdbc:h2:mem:kitchendb;MODE=MySQL;DATABASE_TO_LOWER=TRUE;CASE_INSENSITIVE_IDENTIFIERS=TRUE`，用户名 `sa`，密码为空。
- 本服务 **没有 data.sql 种子数据**（会话/计时器/语音日志均为运行时产生）。

### 2.2 uat / prod（MySQL 8.0）

```powershell
# 先建库并执行本模块 DDL（表带中文 COMMENT）
mysql -uroot -p < D:\self\food-app\sql\00_database.sql
mysql -uroot -p < D:\self\food-app\sql\03_kitchen.sql
```

- `ddl-auto: none`，连接 `jdbc:mysql://localhost:3306/food_app`，账号密码经环境变量 `DB_USERNAME` / `DB_PASSWORD` 注入（默认 root/root）。
- JWT 密钥必须配置环境变量 `JWT_SECRET`（与用户服务保持一致，否则 Token 校验失败一律 40100）；菜谱服务地址可用 `RECIPE_SERVICE_URL` 覆盖（默认 `http://localhost:8082`）。
- prod 额外：关闭 SQL 打印、根日志 INFO。

## 4. 前置条件：获取 Token

先到用户服务注册（或登录）拿 Token（dev 下两服务共用默认 JWT 密钥）：

```powershell
# PowerShell：注册并把 Token 存入变量（重复执行时改用登录）
$reg = Invoke-RestMethod -Uri "http://localhost:8081/api/user/register" -Method Post -ContentType "application/json;charset=utf-8" -Body (@{username="kitchen_tester";password="Test@1234";nickname="厨房测试员"} | ConvertTo-Json)
$token = $reg.data.token

# 已注册过则登录
$login = Invoke-RestMethod -Uri "http://localhost:8081/api/user/login" -Method Post -ContentType "application/json;charset=utf-8" -Body (@{username="kitchen_tester";password="Test@1234"} | ConvertTo-Json)
$token = $login.data.token

# 统一请求头
$H = @{ Authorization = "Bearer $token" }
$token   # 确认非空
```

curl.exe 等价命令：

```powershell
curl.exe -s -X POST http://localhost:8081/api/user/register -H "Content-Type: application/json" -d "{\"username\":\"kitchen_tester\",\"password\":\"Test@1234\",\"nickname\":\"厨房测试员\"}"
# 从响应 data.token 复制 Token，后续命令用 -H "Authorization: Bearer <token>"
```

> 测试用菜谱建议使用菜谱服务种子数据 **recipeId=1（宫保鸡丁，8 步：PREPARE×2 → WASH×1 → CUT×1 → COOK×3 → PLATE×1）**。其中 COOK 第1步「滑炒鸡丁」`timerSec=120`，适合语音"计时"测试。

## 5. 接口测试用例（共 10 个接口）

统一响应体：`{"code":0,"message":"...","data":...,"timestamp":"..."}`。以下命令均假设已设置 `$H`。

### 5.1 POST /api/kitchen/session —— 开始烹饪

```powershell
$s = Invoke-RestMethod -Uri "http://localhost:8083/api/kitchen/session" -Method Post -Headers $H -ContentType "application/json;charset=utf-8" -Body (@{recipeId=1} | ConvertTo-Json)
$s | ConvertTo-Json -Depth 6
$sid = $s.data.id
```

预期：code=0，message=`开始烹饪`，data 为 SessionVO：

```json
{
  "id": 1, "recipeId": 1, "recipeName": "宫保鸡丁",
  "currentPhase": "PREPARE", "currentStepIndex": 1,
  "status": 1, "progressPercent": 0,
  "currentStep": {"id":1,"recipeId":1,"phase":"PREPARE","stepIndex":1,"actionTitle":"食材称重与腌制","detail":"...","mediaUrl":"...","timerSec":900,"firePower":null},
  "steps": ["...共8条，按 PREPARE→PLATE 排序"],
  "timers": [], "startedAt": "2026-06-11 10:00:00", "finishedAt": null
}
```

### 5.2 GET /api/kitchen/session/{id} —— 查询会话

```powershell
Invoke-RestMethod -Uri "http://localhost:8083/api/kitchen/session/$sid" -Headers $H | ConvertTo-Json -Depth 6
```

预期：code=0，data 结构同上（含全量 steps、currentStep、timers、progressPercent）。

### 5.3 POST /api/kitchen/session/{id}/next —— 下一步

```powershell
Invoke-RestMethod -Uri "http://localhost:8083/api/kitchen/session/$sid/next" -Method Post -Headers $H | ConvertTo-Json -Depth 4
```

预期：code=0；阶段内推进 stepIndex+1；阶段末自动进入下一个**有步骤的**阶段第1步；PLATE 最后一步再 next → `status=2`、`finishedAt` 写入、`progressPercent=100`。

以菜谱1为例的进度对照表（可逐次 next 验证）：

| 当前位置 | progressPercent |
|---|---|
| PREPARE.1（初始） | 0 |
| PREPARE.2 | 5 |
| WASH.1 | 10 |
| CUT.1 | 15 |
| COOK.1 | 30 |
| COOK.2 | 50 |
| COOK.3 | 70 |
| PLATE.1 | 90 |
| 再 next → 完成 | 100（status=2） |

### 5.4 POST /api/kitchen/session/{id}/prev —— 上一步（仅阶段内）

```powershell
Invoke-RestMethod -Uri "http://localhost:8083/api/kitchen/session/$sid/prev" -Method Post -Headers $H | ConvertTo-Json -Depth 4
```

预期：当前 stepIndex>1 时回退一步（code=0）；**已在阶段第1步再 prev → code=40900，message=`烹饪阶段不可逆向跳跃`**（见 6.1）。

### 5.5 POST /api/kitchen/session/{id}/timer —— 开计时器（支持多个并行）

```powershell
$t1 = Invoke-RestMethod -Uri "http://localhost:8083/api/kitchen/session/$sid/timer" -Method Post -Headers $H -ContentType "application/json;charset=utf-8" -Body (@{timerName="焯水";totalSec=60} | ConvertTo-Json)
$t2 = Invoke-RestMethod -Uri "http://localhost:8083/api/kitchen/session/$sid/timer" -Method Post -Headers $H -ContentType "application/json;charset=utf-8" -Body (@{timerName="焖煮";totalSec=300} | ConvertTo-Json)
$t1.data; $t2.data
$timerId = $t2.data.id
```

预期：code=0，message=`计时开始`，data 为 TimerVO `{id,timerName,totalSec,remainSec,status:1}`，remainSec≈totalSec。参数约束：timerName 必填且 ≤50 字符；totalSec 必填，1~86400。

### 5.6 GET /api/kitchen/session/{id}/timer —— 计时器列表（到点自动置完成）

```powershell
Invoke-RestMethod -Uri "http://localhost:8083/api/kitchen/session/$sid/timer" -Headers $H | ConvertTo-Json -Depth 4
```

预期：code=0，data 为 `[TimerVO]`，remainSec 实时递减；超过 totalSec 后再查询，对应计时器 `status=2`（已完成）、`remainSec=0`。即使不查询，后台 `@Scheduled` 每 5 秒也会兜底把到点计时器置为已完成。

### 5.7 POST /api/kitchen/timer/{timerId}/cancel —— 取消计时器

```powershell
Invoke-RestMethod -Uri "http://localhost:8083/api/kitchen/timer/$timerId/cancel" -Method Post -Headers $H
```

预期：code=0，message=`计时器已取消`；再查列表该计时器 `status=3`、`remainSec=0`。仅"计时中"可取消（见 6.6）。

### 5.8 POST /api/kitchen/session/{id}/voice —— 语音指令

```powershell
Invoke-RestMethod -Uri "http://localhost:8083/api/kitchen/session/$sid/voice" -Method Post -Headers $H -ContentType "application/json;charset=utf-8" -Body (@{commandText="下一步"} | ConvertTo-Json) | ConvertTo-Json -Depth 6
```

解析规则与预期（按优先级匹配，每条指令无论成败均落库 `kitchen_voice_log`）：

| 指令文本示例 | parsedAction | 预期 message / 行为 |
|---|---|---|
| `下一步` / `继续` | NEXT_STEP | `已进入下一步：<动作标题>`，返回最新 session；走完全部步骤时 message=`全部步骤已完成，烹饪结束，开吃吧！` 且 session.status=2 |
| `上一步` / `回退` | PREV_STEP | 阶段内回退成功 `已回到上一步：<动作标题>`；阶段第1步触发 **40900**（HTTP 响应整体为 40900，语音日志仍已落库） |
| `还有多久` / `几分钟` | QUERY_TIMER | 有计时器：`焖煮还剩4分30秒`（多个用中文逗号拼接）；无计时器：`当前没有进行中的计时器`（session 为 null） |
| `计时` | START_TIMER | 当前步骤 timerSec>0：以步骤标题为名开计时，如 COOK.1 时 `已为「滑炒鸡丁」开启2分钟计时`，session.timers 含新计时器；timerSec=0/null：`当前步骤无需计时` |
| `今天天气怎么样` | UNKNOWN | `未能识别指令`，session 为 null |

### 5.9 POST /api/kitchen/session/{id}/finish —— 主动完成

```powershell
Invoke-RestMethod -Uri "http://localhost:8083/api/kitchen/session/$sid/finish" -Method Post -Headers $H | ConvertTo-Json -Depth 4
```

预期：code=0，message=`烹饪完成`，data.status=2、finishedAt 非空、progressPercent=100。

### 5.10 POST /api/kitchen/session/{id}/abandon —— 放弃烹饪

```powershell
Invoke-RestMethod -Uri "http://localhost:8083/api/kitchen/session/$sid/abandon" -Method Post -Headers $H
```

预期：code=0，message=`已放弃本次烹饪`，data=null；再查会话 status=3。

### 5.11 全流程串测脚本（开始烹饪→逐步 next→开计时器→语音指令→完成）

保存为 `kitchen-e2e.ps1` 直接执行（需 8081/8082/8083 均已启动）：

```powershell
$ErrorActionPreference = "Stop"
$base = "http://localhost:8083/api/kitchen"

# 0) 登录拿 Token（首次自动注册）
try {
  $auth = Invoke-RestMethod -Uri "http://localhost:8081/api/user/login" -Method Post -ContentType "application/json;charset=utf-8" -Body (@{username="kitchen_tester";password="Test@1234"} | ConvertTo-Json)
} catch {
  $auth = Invoke-RestMethod -Uri "http://localhost:8081/api/user/register" -Method Post -ContentType "application/json;charset=utf-8" -Body (@{username="kitchen_tester";password="Test@1234";nickname="厨房测试员"} | ConvertTo-Json)
}
$H = @{ Authorization = "Bearer $($auth.data.token)" }
Write-Host "Token 获取成功"

# 1) 开始烹饪（宫保鸡丁 recipeId=1）
$s = Invoke-RestMethod -Uri "$base/session" -Method Post -Headers $H -ContentType "application/json;charset=utf-8" -Body '{"recipeId":1}'
$sid = $s.data.id
Write-Host ("1) 会话{0} 创建: {1}.{2} 进度{3}%" -f $sid, $s.data.currentPhase, $s.data.currentStepIndex, $s.data.progressPercent)

# 2) 逐步 next 推进到 COOK 阶段（菜谱1：next 4 次到 COOK.1）
1..4 | ForEach-Object {
  $s = Invoke-RestMethod -Uri "$base/session/$sid/next" -Method Post -Headers $H
  Write-Host ("2.$_) next -> {0}.{1} 进度{2}%" -f $s.data.currentPhase, $s.data.currentStepIndex, $s.data.progressPercent)
}

# 3) 手动开两个并行计时器
$t1 = Invoke-RestMethod -Uri "$base/session/$sid/timer" -Method Post -Headers $H -ContentType "application/json;charset=utf-8" -Body '{"timerName":"焯水","totalSec":10}'
$t2 = Invoke-RestMethod -Uri "$base/session/$sid/timer" -Method Post -Headers $H -ContentType "application/json;charset=utf-8" -Body '{"timerName":"焖煮","totalSec":300}'
Write-Host ("3) 计时器: {0}(剩{1}s), {2}(剩{3}s)" -f $t1.data.timerName,$t1.data.remainSec,$t2.data.timerName,$t2.data.remainSec)

# 4) 语音指令四连测
foreach ($cmd in @("计时","还有多久","下一步","随便说点什么")) {
  $v = Invoke-RestMethod -Uri "$base/session/$sid/voice" -Method Post -Headers $H -ContentType "application/json;charset=utf-8" -Body (@{commandText=$cmd} | ConvertTo-Json)
  Write-Host ("4) 语音'{0}' -> {1}: {2}" -f $cmd, $v.data.parsedAction, $v.data.message)
}

# 5) 等焯水计时器到点后查列表（10秒计时器应自动置已完成 status=2）
Start-Sleep -Seconds 11
$timers = Invoke-RestMethod -Uri "$base/session/$sid/timer" -Headers $H
$timers.data | ForEach-Object { Write-Host ("5) 计时器[{0}]{1}: status={2}, remain={3}s" -f $_.id,$_.timerName,$_.status,$_.remainSec) }

# 6) 异常验证：回退到 COOK.1 后再回退应 40900
try { Invoke-RestMethod -Uri "$base/session/$sid/prev" -Method Post -Headers $H | Out-Null } catch {}
try { Invoke-RestMethod -Uri "$base/session/$sid/prev" -Method Post -Headers $H | Out-Null } catch {}
try {
  Invoke-RestMethod -Uri "$base/session/$sid/prev" -Method Post -Headers $H | Out-Null
} catch { Write-Host ("6) 跨阶段回退被拦截: {0}" -f $_.ErrorDetails.Message) }

# 7) 主动完成
$f = Invoke-RestMethod -Uri "$base/session/$sid/finish" -Method Post -Headers $H
Write-Host ("7) 完成: status={0}, 进度{1}%, finishedAt={2}" -f $f.data.status, $f.data.progressPercent, $f.data.finishedAt)

# 8) 完成后再 next 应 40900
try {
  Invoke-RestMethod -Uri "$base/session/$sid/next" -Method Post -Headers $H | Out-Null
} catch { Write-Host ("8) 已结束会话拒绝操作: {0}" -f $_.ErrorDetails.Message) }
```

> 说明：全局异常处理返回 HTTP 200 + 业务码（40900 等），但鉴权失败为 HTTP 401。`Invoke-RestMethod` 对业务码 40900 不会抛异常（HTTP 仍是 200），上面 try/catch 写法兼容两种情况；若未抛异常，可直接检查返回对象的 `code` 字段。

## 6. 异常场景用例（以实际代码逻辑为准）

| # | 场景 | 操作 | 预期 code / message |
|---|---|---|---|
| 6.1 | **跨阶段回退** | 任一阶段第1步时 POST `/session/{id}/prev` | **40900**`烹饪阶段不可逆向跳跃`；日志输出"已拦截（烹饪阶段不可逆向跳跃）" |
| 6.2 | **会话不属于当前用户** | 用账号 B 的 Token 操作账号 A 的会话 | **40300**`无权操作他人的烹饪会话`（SessionGuard 水平越权防护，计时器接口经所属会话同样校验） |
| 6.3 | 会话不存在 | GET/POST `/session/99999/...` | **40400**`烹饪会话不存在`；取消不存在的计时器 → 40400`计时器不存在` |
| 6.4 | 会话已结束再操作 | finish/abandon 后再 next/prev/finish/开计时器 | **40900**`烹饪会话已结束，无法继续操作`（开计时器为`会话已结束，无法开启计时器`） |
| 6.5 | 菜谱不存在 | POST `/session` body `{"recipeId":99999}` | **40400**`菜谱不存在`（透传下游 40400） |
| 6.6 | 重复取消/取消已完成计时器 | 对 status≠1 的计时器 cancel | **40900**`计时器已结束，无法取消` |
| 6.7 | 菜谱服务未启动 | 停掉 8082 后创建会话 | **50200**`依赖服务调用失败，请稍后重试`（注意：步骤缓存 10 分钟内查询已有会话仍可成功） |
| 6.8 | 未登录 | 任意接口不带 Authorization 头 | HTTP **401**，`{"code":40100,"message":"未登录或登录已过期"}` |
| 6.9 | 参数校验失败 | 创建会话缺 recipeId / 计时器 totalSec=0 或 >86400 / commandText 为空 | **40000**，message 含字段提示，如 `totalSec 计时秒数至少为1秒` |
| 6.10 | **语音 UNKNOWN** | voice body `{"commandText":"播放音乐"}` | code=0，data.parsedAction=`UNKNOWN`，message=`未能识别指令`；`kitchen_voice_log` 仍新增一条 |
| 6.11 | 语音"计时"但步骤无计时配置 | 在 timerSec=0 的步骤（如 PREPARE.2）说"计时" | code=0，message=`当前步骤无需计时`，不创建计时器 |
| 6.12 | 菜谱无步骤 | 对无步骤菜谱创建会话 | **40900**`该菜谱暂无步骤，无法开始烹饪` |

## 9. 已知差异（代码 vs 契约）

1. **创建会话初始阶段**：契约描述初始为 PREPARE 第1步；代码实现为"**第一个有步骤的阶段**第1步"（`firstPhaseWithSteps`，对缺阶段的 UGC 菜谱容错）。官方种子菜谱均有 PREPARE 步骤，表现与契约一致。
2. **语音指令多关键词优先级**：契约只列了关键词规则，未定义同时命中时的优先级；代码按 NEXT_STEP → PREV_STEP → QUERY_TIMER → START_TIMER 顺序匹配，例如"继续上一步"会解析为 NEXT_STEP（"继续"先命中）。
3. **步骤缓存**：契约未提及；代码对 `/api/recipe/{id}/steps` 结果做 10 分钟内存缓存，菜谱步骤变更后厨房会话最长 10 分钟内仍使用旧步骤。
4. 错误码 40300（非本人会话）、40400（会话/菜谱/计时器不存在）、50200（下游失败）为契约总则之外代码实际使用的扩展码，语义与统一约定一致。
