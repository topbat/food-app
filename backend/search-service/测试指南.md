# 智能搜索推荐服务（search-service）测试指南

## 1. 模块简介

| 项目 | 说明 |
|---|---|
| 服务名 | search-service（智能搜索推荐服务） |
| 端口 | **8084** |
| 接口前缀 | `/api/search` |
| 表前缀 | `search_`（search_history / search_hot_keyword） |
| 核心职责 | 多维检索（关键词/食材/营养/人群，透传菜谱服务）、热搜 TOP10（30秒内存缓存）、登录用户搜索历史（最近20条）、个性化推荐（标签映射 + 游客热度降级） |
| 技术 | Spring Boot 3.3.5 + Java 21 + JPA |

**鉴权策略（三种）**：

- `/api/search/history`（GET/DELETE）：**强制登录**（AuthInterceptor 拦截，无 Token 返回 HTTP 401 / 40100）；
- `/api/search/recipes`、`/api/search/recommend`：**可选登录**——Controller 手动解析 Authorization 头，Token 缺失/非法/过期一律按**游客**处理，**不报 401**；
- `/api/search/hot`：完全公开。

**对其他服务的依赖（启动顺序）**：

1. **菜谱服务 :8082（强依赖）**——搜索透传 `GET /api/recipe/list`；推荐取菜谱列表。搜索链路下游失败直接报 **50200**（核心链路）；推荐链路下游失败**降级**为热度策略/空列表，永不报错。
2. **用户服务 :8081（推荐链路依赖 + 发 Token）**——个性化推荐调用 `GET /api/user/tag/list`（原样转发 Authorization 头）；测试登录态需先在用户服务注册/登录。

> 推荐启动顺序：菜谱服务 8082 → 用户服务 8081 → 搜索服务 8084。

## 2. 环境准备

### 2.1 dev（默认，免安装）

- H2 内存库（MySQL 兼容模式），`ddl-auto: update` 自动建表。
- JDBC URL：`jdbc:h2:mem:searchdb;MODE=MySQL;DATABASE_TO_LOWER=TRUE;CASE_INSENSITIVE_IDENTIFIERS=TRUE`，用户名 `sa`，密码为空。
- 本服务**没有 data.sql 种子数据**：刚启动时热搜榜为空数组、搜索历史为空，需要先搜几次"喂"数据。

### 2.2 uat / prod（MySQL 8.0）

```powershell
mysql -uroot -p < D:\self\food-app\sql\00_database.sql
mysql -uroot -p < D:\self\food-app\sql\04_search.sql
```

- `ddl-auto: none`，连接 `jdbc:mysql://localhost:3306/food_app`（`DB_USERNAME`/`DB_PASSWORD`，默认 root/root）。
- 环境变量：`JWT_SECRET`（须与用户服务一致）、`RECIPE_SERVICE_URL`（默认 http://localhost:8082）、`USER_SERVICE_URL`（默认 http://localhost:8081）。
- prod 额外：关闭 SQL 打印、根日志 INFO。

### 2.3 切换环境


## 4. 前置条件：获取 Token（测试登录态功能用）

```powershell
# 注册（已注册过则改用 login）
$reg = Invoke-RestMethod -Uri "http://localhost:8081/api/user/register" -Method Post -ContentType "application/json;charset=utf-8" -Body (@{username="search_tester";password="Test@1234";nickname="搜索测试员"} | ConvertTo-Json)
$token = $reg.data.token
$H = @{ Authorization = "Bearer $token" }

# 给当前用户加一个 Z 世代标签（用于个性化推荐验证；tagType=3 场景标签）
Invoke-RestMethod -Uri "http://localhost:8081/api/user/tag" -Method Post -Headers $H -ContentType "application/json;charset=utf-8" -Body (@{tagName="减脂期";tagType=1} | ConvertTo-Json)
```

curl.exe 等价：

```powershell
curl.exe -s -X POST http://localhost:8081/api/user/login -H "Content-Type: application/json" -d "{\"username\":\"search_tester\",\"password\":\"Test@1234\"}"
```

> 标签 → 菜谱标签映射表（代码内置，影响推荐结果）：`熬夜党→快手菜`、`健身后→高蛋白`、`减脂期→低卡`、`生理期→生理期友好`、`增肌期→高蛋白`；未命中映射的标签按同名匹配。菜谱种子库中带"低卡"标签的有菜谱 7/10 等。

## 5. 接口测试用例（共 5 个接口）

### 5.1 GET /api/search/recipes —— 多维搜索（可选登录）

query 参数：`page`(默认1,≥1)、`size`(默认10,1~50)、`cuisineType`、`difficulty`、`keyword`、`maxCalories`(≥1)、`tagName`、`ingredients`(逗号分隔，冰箱清理模式)、`searchType`(默认1，取值1~4：1关键词 2按食材 3按营养 4按人群)。

```powershell
# ① 游客关键词搜索（热搜计数+1，不写历史）
Invoke-RestMethod -Uri "http://localhost:8084/api/search/recipes?keyword=鸡丁&page=1&size=10" | ConvertTo-Json -Depth 5

# ② 登录用户关键词搜索（热搜计数+1，并写入搜索历史）
Invoke-RestMethod -Uri "http://localhost:8084/api/search/recipes?keyword=宫保&searchType=1" -Headers $H | ConvertTo-Json -Depth 5

# ③ 食材搜索（冰箱清理模式：匹配含任一食材的菜谱）
Invoke-RestMethod -Uri "http://localhost:8084/api/search/recipes?ingredients=鸡胸肉,豆腐&searchType=2" -Headers $H

# ④ 营养目标搜索（最大热量）
Invoke-RestMethod -Uri "http://localhost:8084/api/search/recipes?maxCalories=300&searchType=3" -Headers $H

# ⑤ 人群标签搜索
Invoke-RestMethod -Uri "http://localhost:8084/api/search/recipes?tagName=减脂期&searchType=4" -Headers $H

# ⑥ 组合条件（菜系1=川菜 + 难度2）
Invoke-RestMethod -Uri "http://localhost:8084/api/search/recipes?cuisineType=1&difficulty=2"
```

curl.exe 等价（中文参数需 URL 编码或直接用引号交给 curl 处理）：

```powershell
curl.exe -s -G "http://localhost:8084/api/search/recipes" --data-urlencode "keyword=宫保" -H "Authorization: Bearer %TOKEN%"
```

预期：code=0，data 为菜谱服务分页结构 `{total,page,size,list:[RecipeSummary]}`，RecipeSummary 含 `{id,title,coverUrl,cuisineType,cuisineName,difficulty,totalTimeMin,caloriesKcal,carbsG,proteinG,fatG,description,viewCount,likeCount,tags}`。

附加效果验证：

- keyword 非空且下游成功 → `search_hot_keyword` 中该词 `search_count` +1（重复搜同词累加，upsert 不产生新行）；
- 带合法 Token → `search_history` 新增一条（keyword/searchType/userId）；
- keyword 为空（仅 tagName/ingredients 等条件）→ **不**计热搜、**不**写历史。

### 5.2 GET /api/search/hot —— 热搜 TOP10（公开）

```powershell
# 先制造热度：同一关键词搜 3 次
1..3 | ForEach-Object { Invoke-RestMethod -Uri "http://localhost:8084/api/search/recipes?keyword=麻婆豆腐" | Out-Null }
Invoke-RestMethod -Uri "http://localhost:8084/api/search/hot" | ConvertTo-Json -Depth 4
```

预期：code=0，data 为按 `searchCount` 降序的数组（最多10条）：`[{"keyword":"麻婆豆腐","searchCount":3},...]`。

> 注意：结果有 **30 秒内存缓存**——刚搜完立即查热搜，计数可能未反映，等 30 秒后再查即可（验证缓存行为本身也是一个用例：30 秒内两次请求日志只回源一次）。

### 5.3 GET /api/search/history —— 我的搜索历史（强制登录）

```powershell
Invoke-RestMethod -Uri "http://localhost:8084/api/search/history" -Headers $H | ConvertTo-Json -Depth 4
```

预期：code=0，data 为最近 **20** 条按时间倒序：`[{"id":1,"keyword":"宫保","searchType":1,"createdAt":"2026-06-11 10:00:00"}]`（不暴露 userId）。

### 5.4 DELETE /api/search/history —— 清空历史（强制登录）

```powershell
Invoke-RestMethod -Uri "http://localhost:8084/api/search/history" -Method Delete -Headers $H
# 预期：{"code":0,"message":"搜索历史已清空","data":null}；再查 5.3 返回空数组
```

### 5.5 GET /api/search/recommend —— 个性化推荐（可选登录）

```powershell
# ① 游客：按热度推荐
Invoke-RestMethod -Uri "http://localhost:8084/api/search/recommend?limit=5" | ConvertTo-Json -Depth 5
# 预期 data.reason = "大家都在做的高热度菜谱"，list 为热度前5的 RecipeSummary

# ② 登录 + 已加"减脂期"标签：个性化推荐
Invoke-RestMethod -Uri "http://localhost:8084/api/search/recommend?limit=10" -Headers $H | ConvertTo-Json -Depth 5
# 预期 data.reason = "根据你的标签【减脂期】为你推荐"，list 为映射标签"低卡"命中的菜谱（每个映射标签取5条候选，按菜谱ID聚合去重后截取 limit 条）

# ③ 登录但无任何标签：回落游客策略（可用新注册未加标签的账号验证）
# 预期 reason 同游客文案
```

参数约束：`limit` 默认 10，1~50。

## 6. 异常场景用例（以实际代码逻辑为准）

| # | 场景 | 操作 | 预期 |
|---|---|---|---|
| 6.1 | 未登录查历史 | GET/DELETE `/history` 不带 Token | HTTP **401**，`{"code":40100,"message":"未登录或登录已过期"}` |
| 6.2 | **非法 Token 走可选登录接口** | `/recipes` 或 `/recommend` 带 `Authorization: Bearer 伪造串` | **code=0 正常返回**（按游客降级，不写历史/不个性化）；日志 WARN `[可选登录] Token解析失败, 按游客处理` |
| 6.3 | 参数越界 | `?page=0`、`?size=51`、`?searchType=5`、`?maxCalories=0`、`?limit=0` | **40000**，message 为约束提示（如 `searchRecipes.searchType: 搜索类型取值1-4`） |
| 6.4 | **菜谱服务未启动时搜索** | 停掉 8082 后 GET `/recipes?keyword=x` | **50200**`菜谱服务调用失败，请稍后重试`（核心链路直接报错）；且该次**不**计热搜、不写历史 |
| 6.5 | **菜谱服务未启动时推荐** | 停掉 8082 后 GET `/recommend` | **code=0**，reason=`大家都在做的高热度菜谱`，`list=[]`（兜底降级，永不报错）；日志 WARN `推荐降级` |
| 6.6 | **用户服务未启动时登录推荐** | 停掉 8081 后带 Token GET `/recommend` | **code=0**，降级为游客热度策略；日志 WARN `[推荐] 推荐降级: 标签个性化链路失败...` |
| 6.7 | 登录用户无标签推荐 | 新账号（未加标签）GET `/recommend` | code=0，游客文案；日志 `[推荐] 用户无标签, 回落游客策略` |
| 6.8 | 清空空历史 | 历史为空时 DELETE `/history` | code=0（清除 0 条，幂等） |
| 6.9 | keyword 全空格 | `/recipes?keyword=%20%20` | 正常透传查询，不计热搜（isBlank 判断） |

## 8. 数据库验证点（dev / H2）

H2 Console：`http://localhost:8084/h2-console`，JDBC URL 填 `jdbc:h2:mem:searchdb`，用户名 `sa`，密码留空。

```sql
-- 热搜：同一关键词只有一行，search_count 随搜索累加（uk_keyword 唯一约束）
SELECT id, keyword, search_count, updated_at FROM search_hot_keyword ORDER BY search_count DESC;

-- 搜索历史：仅登录用户的搜索产生记录；游客搜索不应出现在此表
SELECT id, user_id, keyword, search_type, created_at FROM search_history ORDER BY created_at DESC;

-- 验证“最近20条”窗口：插入超过20条后接口只返回20条，但表中保留全部
SELECT COUNT(*) FROM search_history WHERE user_id = 1;

-- 清空验证：DELETE /history 后该用户记录数应为 0
SELECT COUNT(*) FROM search_history WHERE user_id = 1;
```

uat/prod 在 MySQL `food_app` 库执行同样 SQL。H2 为内存库，**服务重启后热搜/历史清空**。

## 9. 已知差异（代码 vs 契约）

1. **热搜计数时机**：契约写"keyword 非空时热搜计数+1"；代码实现为**仅当菜谱服务调用成功后**才计数与写历史（下游失败的搜索不计入热搜/历史）。
2. **searchType 不透传下游**：`searchType` 仅做参数校验（1~4）并写入搜索历史，不会作为查询条件传给菜谱服务（菜谱服务 `/api/recipe/list` 本身无该参数，符合契约"query 同 /api/recipe/list 另加 searchType"的本意，特此说明）。
3. **热搜 30 秒缓存**：契约未提及；新计数最长延迟 30 秒才体现在 `/hot` 榜单（单机内存缓存，多实例部署时各实例独立）。
4. **推荐接口永不报错**：契约只写了两种策略；代码额外实现了三级降级（个性化失败→热度策略→空列表），下游全部不可用时返回 code=0 + 空 list，而非 50200。
5. 推荐接口带**非法/过期 Token** 时按游客处理返回 200，与契约"否(可带token)"语义一致但契约未明确说明。
